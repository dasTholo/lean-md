# lmd-dispatching-parallel-agents — Port Implementation Plan

Spec: `docs/lean-md/specs/2026-07-06-lmd-dispatching-parallel-agents-design.md`.

## Goal

Port the superpowers `dispatching-parallel-agents` skill to a native lmd skill
`lmd-dispatching-parallel-agents` (full replacement, no functional loss) AND anchor the
same capability as an opt-in mode inside `lmd-subagent-driven-development` (SDD). Both
consumers share ONE reusable fragment as single source (`@include parallel-dispatch`).

## Architecture (ambient repo plumbing — stated once, not per task)

Single source + two consumers (fragment-consistency-gate pattern — precedent:
`test-first-core`, `skill-authoring-core`, `brainstorm-gate`):

- `content/core/_fragments/parallel-dispatch.lmd.md` — seed (built-in via `include_str!`
  in `src/fragments.rs`; on-disk-consistency gate: built-in const == on-disk seed).
- Consumer A — standalone skill `content/skills/lmd-dispatching-parallel-agents/{SKILL.md,
  body.lmd.md}` (4 phases pre-context → assess → dispatch → integrate; `@include
  parallel-dispatch` in assess + dispatch). Three registration surfaces, identical shape to
  the existing skills: `src/skills.rs` (`const … _BODY` + `SKILLS` row — NO `COMPANIONS`),
  `src/skill_install.rs` (`const … _SKILL_MD` + `INSTALLABLE_SKILLS` — NO `ASSETS`),
  `src/availability.rs` (`COVERAGE` rows, coverage gate).
- Consumer B — SDD body edit: `preflight` `next:` → `dispatch-mode`; new gate phase
  `dispatch-mode` (sequential vs. parallel query) + new phase `parallel-dispatch`
  (`@include parallel-dispatch`, reuses the existing `implementer` companion; `@call
  task_return` reused 1:1 per returning agent). The SDD body const auto-syncs via the
  existing `include_str!` (`LMD_SDD_BODY`) — no new body const.

On-disk seed == `include_str!` target (the content file IS the seed), so the
fragment-consistency invariant holds trivially. The root `.claude/skills/<name>/` stub is
install-generated by `install_skill` (`INSTALLABLE_SKILLS` drives it) — NOT git-tracked,
NOT authored by hand.

**Seed content is already in place under `content/` (its real home).** The three new
additive seeds already exist in the working tree — the plan author wrote them there, NOT into
a `plans/assets/` shadow copy:
- `content/core/_fragments/parallel-dispatch.lmd.md`
- `content/skills/lmd-dispatching-parallel-agents/body.lmd.md`
- `content/skills/lmd-dispatching-parallel-agents/SKILL.md`

Their tasks do NOT recreate them — each task verifies the file is present, adds the Rust
registration (that is where TDD-RED bites: an unregistered seed → `resolve`/`skill_body`/
`install_skill` fail), and commits the seed together with its wiring. Rationale for NOT
inlining seed text into this plan: seed content carries `@phase`/`@include`/`@query`
directives; inlining it makes the plan renderer segment/execute them (confirmed — an inlined
`@phase` truncates the surrounding task; a fenced block only strips the `@phase` line from the
render). So the seed lives as a real file, never inline.

**One exception — the SDD body EDIT is staged as an asset:**
`docs/lean-md/plans/assets/2026-07-06-lmd-dispatching-parallel-agents-sdd-body.lmd.md`. It is a
byte-identical replacement of the *already-registered, live* SDD body (renders as 8 phases:
orient · preflight · dispatch-mode · dispatch · review · parallel-dispatch · final-review ·
handoff — only `preflight next:` changed + the two new phases inserted, every other phase
byte-for-byte unchanged). It is staged (not pre-applied to the live file) so Task 4 keeps a
real RED→GREEN: the new phases must be absent until the task runs. Copy it into the live body
with `ctx_read` (raw lmd source) or `lean-md source <asset>`, written verbatim (#498).

## Global Constraints

- Non-goals (YAGNI): no worktree isolation; no new coordination infra (no `ctx_share`, no SDD
  bash scripts); no weakening of the two-verdict per-task review — only execution fans out. The
  `dispatch` phase of SDD stays sequential and byte-identical to today.
- Single source: exactly ONE fragment `parallel-dispatch`; both consumers `@include` it — the
  fan-out rule / prompt-structure / memory-coordination block are never duplicated in a body.
- Non-goal: do NOT modify the built-in `dispatch-contract` fragment or the `@dispatch` bridge
  (both SDD and this port depend on it unchanged).
- #498 determinism (test gate): the fragment-consistency gate (built-in const == on-disk seed)
  stays green; no timestamps/counters/random in any seed.
- Reference-closure (test gate): neither the standalone `SKILL.md` nor `body.lmd.md` may carry
  the token `superpowers` — asserted per-seed in Task 2's render test + Task 3's install test.
- Frontmatter-scalar guard (test gate): the standalone `SKILL.md` `description` is a single-line,
  non-empty scalar, unquoted-and-free-of `": "` (or quoted).
- Mandatory memory/coordination block lives in the fragment so BOTH consumers (incl. the
  standalone skill) inherit the `ctx_session`/`ctx_knowledge`/`ctx_agent` discipline.
- Task dependencies: Task 1 (fragment) precedes Task 2 and Task 4 (both `@include
  parallel-dispatch`; an unresolved include renders `<!-- @include err -->` and fails the
  fragment-marker assertions). Task 2 precedes Task 3 (SKILL.md must be registered). Tasks 2 + 4
  precede Task 5 (COVERAGE asserts the phases exist).
